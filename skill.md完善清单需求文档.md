# skill.md完善清单需求文档

## 1. 文档目的

本文档用于重新评估 `annual-report-extractor` 项目的 `SKILL.md`，明确后续完善方向应从“重型运行手册”调整为：

> **让 `SKILL.md` 保持精简、稳定、易读，把复杂操作、排障闭环、验证闭环尽量下沉到代码中实现。**

本次调整采用以下前提：

1. `SKILL.md` 的首要职责不是承载全部操作细节，而是提供**最小必要导航**。
2. 凡是可以被脚本自动执行、自动判断、自动诊断的内容，应优先放到 `scripts/` 中，而不是堆积在文档里。
3. 文档的价值应集中在：
   - 告诉执行者先做什么
   - 告诉执行者失败时先跑哪个入口
   - 告诉执行者交付完成的最低标准是什么
4. 排障规则、验证规则、结构检查、落盘检查，原则上应逐步产品化为脚本能力。

因此，本次需求文档不再把“更长、更细、更像手册”视为默认优化方向，而是重点评估：

- `SKILL.md` 是否足够精简清晰
- 主链路是否能一眼看懂
- 文档是否避免承担过多本应由代码完成的判断责任
- 复杂规则是否已经或应该下沉为脚本能力
- 文档与代码是否更容易长期同步维护

## 3. 对 skill 文档定位的重新确认

### 3.1 新定位

结合项目现状与维护目标，`SKILL.md` 更适合被定义为：

> 一个面向执行者的精简主入口文档，用来说明技能目标、推荐主链路、核心命令、最小验收标准，以及在失败时应该优先进入哪个诊断入口。

### 3.2 在该定位下，`SKILL.md` 应重点承担的职责

`SKILL.md` 应重点保留以下内容：

1. 技能目标与输出目标
2. 环境前提与最小检查命令
3. 脚本速查
4. 固定主链路顺序
5. 最终双表交付标准
6. 出错时优先执行哪个脚本

### 3.3 在该定位下，不宜继续在 `SKILL.md` 中膨胀的内容

以下内容不是不能存在，而是**不应继续优先堆在 `SKILL.md` 主文档中**：

- 大段 AKShare 故障分类文本
- 大段人工判读说明
- 复杂验证优先级规则的大篇幅叙述
- 过多 pandas / openpyxl 泛化示例
- 过细的手工回读和人工诊断步骤
- 大量“如果 A 则 B，否则 C”的文档化分支判断

这些内容更适合：

1. 下沉到脚本中自动执行
2. 放入独立参考文档
3. 作为诊断工具的输出结果体现

### 3.4 在该定位下，真正需要关注的质量标准

精简型 `SKILL.md` 的核心质量标准不是“信息多”，而是以下六项：

1. **可执行性**：命令、路径、参数、脚本名都真实可用。
2. **最小清晰度**：执行者能快速理解先做什么、后做什么。
3. **主链路聚焦**：内容围绕“获取更多正确的权威信息和稳定生成双表交付”。
4. **代码优先**：能程序化的规则尽量交给脚本，而不是交给阅读者。
5. **可维护性**：文档不因过长过细而快速失效。
6. **与代码对齐**：文档内容与当前仓库结构、脚本接口、依赖状态保持一致。

## 4. 总体判断

当前 `SKILL.md` 已具备较完整的信息密度，但如果后续目标是“更稳定、更容易维护、更不容易过期”，则继续把它强化为重型运行手册并不是最优路线。

当前更合理的方向应是：

1. `SKILL.md` 保留为**精简主链路文档**。
2. 把复杂操作和复杂判断尽量下沉到脚本。
3. 把排障闭环从“文字规则”升级为“诊断脚本输出”。
4. 把验证闭环从“手工核对清单”升级为“自动检查脚本”。
5. 让文档负责导航，让代码负责执行和判断。

所以，本次完善方向应调整为：

> 不继续把 `SKILL.md` 扩写为重型操作手册；而是让 `SKILL.md` 更精简，把复杂操作、排障闭环、验证闭环逐步沉到代码里。

## 5. 主要问题清单

### 5.1 `SKILL.md` 容易承担过多本应由代码完成的说明责任

#### 问题表现

当前思路中，很多复杂内容都倾向继续写进 `SKILL.md`，例如：

- AKShare 故障分类
- 字段变化判读
- 验证失败优先级
- 落盘确认步骤
- 结构检查细则
- 辅助示例与操作细则

这些内容虽然有价值，但如果都堆在主文档中，会导致：

- 文档越来越长
- 读者执行成本越来越高
- 文档与代码越来越难同步
- 本来可以自动判断的逻辑被迫变成人工判断

#### 风险

- `SKILL.md` 会从“入口文档”变成“维护负担源”。
- 执行者必须读大量文字，才能完成本来可自动化的动作。
- 文档一旦过期，误导性会比简短文档更强。

#### 完善要求

1. 明确 `SKILL.md` 只承担主链路导航职责。
2. 凡是可以程序化的复杂规则，优先转成脚本功能。
3. `SKILL.md` 不再以“覆盖所有异常分支”为目标。
4. 对复杂规则，只保留“入口说明 + 跳转到脚本”的最小表达。

#### 验收标准

- `SKILL.md` 的主阅读目标是“快速开始并知道下一步去哪”。
- 执行者无需阅读大量规则文本，仍能进入正确流程。

---

### 5.2 路径、目录与执行示例仍必须贴近当前仓库真实结构

#### 问题表现

即使 `SKILL.md` 变精简，命令口径仍必须真实。例如：

```bash
python scripts/check_env.py
python scripts/akshare_fetch.py 600660 2025 --format json
python scripts/extract_data.py --company=福耀玻璃 --code=600660.SH --year=2025
```

文档不应继续混用与当前仓库不一致的路径表达。

#### 风险

- 文档再精简，只要命令不真实，仍然会直接误导执行者。
- 精简文档如果命令不准，反而更难补救。

#### 完善要求

1. 所有保留在 `SKILL.md` 的命令，都必须可在当前仓库直接执行。
2. 若需要区分“源码仓库运行方式”和“安装目录运行方式”，要明确标注。
3. 默认优先展示当前仓库可直接复制运行的命令。

#### 验收标准

- 新执行者在项目根目录可直接照抄核心命令。
- 主链路命令不再与当前仓库结构冲突。

---

### 5.3 环境说明应精简且真实，不做过强承诺

#### 问题表现

`SKILL.md` 若过度强调“全部内置、无需安装”，会与 `scripts/check_env.py`、`scripts/requirements.txt` 的现实逻辑冲突。

#### 风险

- 执行者预期错误
- 环境问题会被误判成代码故障
- 精简文档反而失去可信度

#### 完善要求

1. 环境说明只保留真实口径：
   - 仓库已包含脚本源码
   - 运行仍依赖 Python / Node / 已安装依赖
2. 明确：
   - 脚本已包含 ≠ 依赖已安装
3. 推荐把依赖状态检查交给 `check_env.py`，而不是在文档里长篇解释。

#### 验收标准

- 文档环境说明与 `check_env.py` 逻辑自洽。
- 文档不再对所有场景做绝对承诺。

---

### 5.4 主链路应保留，但说明要更短、更固定

#### 问题表现

主链路是 `SKILL.md` 中最值得保留的部分，但不应被扩写成大量手工判断说明。

#### 完善要求

新增或保留一节：`稳定运行最小闭环（推荐执行顺序）`，固定为：

### 推荐顺序
1. `python scripts/check_env.py`
2. 搜索确认年报是否正式发布
3. `python scripts/akshare_fetch.py ...` 做接口探测
4. `python scripts/extract_data.py ...` 跑主链路
5. 必要时执行手动补齐
6. 生成表1
7. 生成表2
8. 执行验证和回读
9. 检查文件是否真实落盘

但每一步应只保留最小必要说明，不在 `SKILL.md` 中展开过多分支规则。

#### 验收标准

- 不看全文，只看该章节也能知道执行顺序。
- 主链路章节简短、稳定、可直接使用。

---

### 5.5 复杂排障应尽量代码化，而不是继续文档化

#### 问题表现

AKShare、接口、字段映射、网络异常、市场识别等问题确实需要处理，但这类逻辑如果主要依赖文档大段描述，会越来越难维护。

#### 更合理的方向

应优先把以下能力做成脚本：

- 环境检查
- AKShare 探测
- 输出结果摘要
- 故障类型归类
- 下一步动作建议

例如未来可考虑统一诊断入口：

```bash
python scripts/run_pipeline_check.py --company=福耀玻璃 --code=600660.SH --year=2025
```

由脚本自动输出：

```text
诊断结果：AKShare 可访问，但主流程未消费到字段
建议动作：检查字段映射，不修改主流程入口
```

#### 完善要求

1. `SKILL.md` 只保留“遇到 AKShare 问题，先跑哪个诊断脚本”。
2. 故障分类、输出判读、建议动作，优先转成脚本输出。
3. 文档中不再要求大量人工阅读后再做判断。

#### 验收标准

- 排障入口更明确。
- 复杂判读规则逐步从文档迁移到代码。

---

### 5.6 验证闭环应尽量脚本化，而不是主要靠手工清单

#### 问题表现

当前关于验证、结构检查、落盘确认的想法多数以文字清单表达，但这些动作本质上很适合自动化。

例如：

- 表1是否存在
- 表2是否存在
- 表1是否为 22 项
- 表2是否为 27 项
- 表1是否包含 `数据验证` Sheet
- 文件是否可回读

这些都更适合由脚本直接判断。

#### 完善要求

1. `SKILL.md` 只保留最小验收标准和核心验证命令。
2. 将结构检查、回读检查、文件存在性检查、验证结果归纳尽量下沉到脚本。
3. 鼓励新增或增强统一验收脚本，而不是持续扩写人工检查说明。

#### 验收标准

- 交付判断越来越依赖脚本结果，而不是依赖人工解读文档。
- `SKILL.md` 中的验证章节明显收敛。

---

### 5.7 示例代码应收敛，优先保留主链路必需示例

#### 问题表现

`SKILL.md` 中如果保留太多 pandas / openpyxl 通用示例，会稀释主链路重点。

#### 完善要求

1. 主文档中优先保留：
   - 环境检查
   - AKShare 诊断
   - 主流程执行
   - 验证 / 回读
2. 其余通用示例：
   - 移到附录
   - 移到 `references/`
   - 或删除重复价值不高的部分

#### 验收标准

- 执行者一眼能看到主链路命令。
- 文档不再因泛化示例过多而分散注意力。

---

### 5.8 文档维护原则应从“补更多说明”转向“补更多脚本能力”

#### 问题表现

如果后续每次发现新异常，都优先往 `SKILL.md` 里补文字说明，文档会越来越重。

#### 完善要求

新增维护原则：

1. 修改 CLI 参数时，必须同步更新 `SKILL.md` 中对应命令。
2. 修改主流程时，必须检查 `SKILL.md` 主链路是否仍成立。
3. 修改验证逻辑时，优先补脚本能力，再决定是否需要补文档说明。
4. 出现新故障场景时，优先考虑新增诊断逻辑，而不是只补长段文字。
5. 能由脚本自动判断的，不优先用文档说明替代。

#### 验收标准

- 文档与脚本同步更新。
- 新问题优先沉淀为代码能力，而不是文档膨胀。

## 7. 建议的新结构方向

在新的定位下，`SKILL.md` 建议更偏向“首页导航式结构”，例如：

1. 技能简介与触发范围
2. 环境前提与检查命令
3. 脚本速查
4. 稳定运行最小闭环
5. 核心交付标准
6. 失败时优先进入的诊断入口
7. 更新日志

而以下内容建议弱化、迁移或代码化：

- 详细 AKShare 排障规则
- 详细验证优先级规则
- 详细回读与落盘检查步骤
- 大量 pandas / openpyxl 通用示例
- 复杂人工决策分支

这个结构不是为了减少质量要求，而是为了让：

- 文档更短
- 导航更清楚
- 执行负担更小
- 自动化空间更大
- 文档与代码更容易长期同步

## 8. 验收标准

当 `SKILL.md` 根据本文档完成完善后，应达到以下验收标准：

### 8.1 可直接执行
- 新执行者按文档能在当前仓库中直接跑通主命令。
- 命令、参数、路径与真实项目一致。

### 8.2 可快速开始
- 不看全文，只看主链路章节就能知道先做什么。
- 核心命令集中、清晰、可复制。

### 8.3 文档不过度承担判断逻辑
- 排障、结构检查、回读检查、落盘检查等复杂内容不再主要依赖大段文字说明。
- 文档更多负责“告诉你去跑哪个脚本”。

### 8.4 可稳定交付
- 即使部分外部源失败，主文档仍能指导完成最小双表交付路径。
- 缺失项有规范标注，失败时有明确诊断入口。

### 8.5 可维护
- 脚本参数、命令、示例变更时，有同步更新要求。
- 新异常优先沉淀为代码能力，而不是无限扩写主文档。

## 9. 推荐实施顺序

### 第一阶段：先修文档定位与真实可执行性
1. 明确 `SKILL.md` 走精简路线
2. 修正路径和命令
3. 修正环境依赖描述
4. 固定主链路 9 步
