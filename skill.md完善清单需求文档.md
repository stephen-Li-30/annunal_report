# skill.md完善清单需求文档

## 1. 文档目的

本文档用于重新评估 `annual-report-extractor` 项目的 `SKILL.md`，明确其作为**运行手册型技能主文档**时，当前仍需要完善的内容。

本次分析采用以下前提：

1. `SKILL.md` 不是简短说明页，而是**脚本使用手册 + 排障手册 + 生成效果保障手册**。
2. `SKILL.md` 可以包含较多示例代码、诊断细节和实现建议。
3. `SKILL.md` 的核心目标不是“精简”，而是：
   - 让技能包更稳定运行
   - 让数据提取链路更稳定
   - 让双表输出更稳定落地
   - 让失败场景更容易排查和补救

因此，本次文档不再将“内容长”“示例多”“诊断细节多”“实现建议多”视为问题本身，而是重点评估：**这些内容是否真实、准确、可执行、可诊断、可维护，并真正有助于稳定生成效果。**

---

## 2. 分析对象

本次分析基于以下对象：

- `SKILL.md`
- `改造清单需求文档.md`
- 当前项目目录结构
- 当前 `scripts/`、`references/`、`examples/outputs/` 的实际存在情况

---

## 3. 对 skill 文档定位的确认

### 3.1 当前定位

结合项目现状与使用目标，`SKILL.md` 应被定义为：

> 一个面向执行者的高密度运行手册，用来指导年报数据提取、数据补齐、Excel生成、结果验证、异常排查与生成补救，最终确保技能包尽可能稳定地产出目标结果。

### 3.2 在该定位下，以下内容属于合理设计

以下内容不应被视为问题，而应视为技能包稳定运行所需的组成部分：

- 较长的使用说明
- 较多脚本示例
- 较多 AKShare / Excel / pandas / 排障示例
- 失败场景分类说明
- 容错流程和替代方案
- 实现建议与操作建议
- 对输出结果的严格验收约束

### 3.3 在该定位下，真正需要关注的质量标准

运行手册型 `SKILL.md` 的核心质量标准不是“短”，而是以下六项：

1. **可执行性**：命令、路径、参数、脚本名都真实可用。
2. **一致性**：相同规则、相同流程、相同术语不冲突。
3. **稳定性导向**：文档真的能帮助产出双表，而不是只堆要求。
4. **可诊断性**：失败时能快速定位问题并给出补救路径。
5. **可维护性**：规则虽多，但更新时不容易失控。
6. **与代码对齐**：文档内容与当前仓库结构、脚本接口、依赖状态保持一致。

---

## 4. 总体判断

当前 `SKILL.md` 已经具备较强的“运行手册”雏形，优点很明显：

- 有完整业务目标
- 有环境说明
- 有脚本速查
- 有快速开始命令
- 有标准流程
- 有异常处理思路
- 有双表输出约束
- 有数据验证要求
- 有示例代码和操作建议

但如果目标是“让技能包稳定运行、稳定生成效果”，当前 `SKILL.md` 仍存在一批需要完善的关键问题。

这些问题不在于“写得太多”，而在于：

1. **有些内容虽然详细，但和当前项目实际不完全对齐。**
2. **有些规则虽然严格，但还不够可执行或可降级。**
3. **有些示例虽然丰富，但未完全形成稳定运行所需的决策顺序。**
4. **有些诊断内容虽然出现了，但还没有形成真正高效的排障闭环。**
5. **有些“必须生成”的要求很强，但缺少对极端失败场景的现实化补救说明。**

所以，本次完善方向应是：

> 保留 `SKILL.md` 作为重型运行手册的定位，不做“去手册化”收缩；但必须增强其真实性、可执行性、诊断闭环、异常分流和稳定交付能力。

## 5. 主要问题清单

### 5.1 路径、目录与执行示例和当前仓库实际存在偏差

#### 问题表现

`SKILL.md` 中多次使用如下命令：

```bash
cd skills/annual-report-extractor/scripts
python check_env.py
```

但当前仓库实际结构是项目根目录下直接存在：

- `SKILL.md`
- `scripts/`
- `references/`
- `examples/`

这意味着文档中的某些路径写法更像“被安装后的技能目录”，而不是当前仓库内的真实执行路径。

同时，依赖安装命令和脚本执行命令有时也未完全说明应在什么目录下运行，例如：

- `pip install -r requirements.txt`
- `python check_env.py`

在当前仓库中，更容易被直接理解为：

- `pip install -r scripts/requirements.txt`
- `python scripts/check_env.py`

#### 风险

- 新执行者按文档直接复制命令时，可能在当前仓库中执行失败。
- 使用者无法分辨“技能安装态路径”和“源码仓库路径”。
- 文档虽详细，但第一步就可能因路径不一致而失败。

#### 完善要求

1. 所有命令示例必须区分两种运行场景：
   - **源码仓库运行方式**
   - **技能安装目录运行方式**（如果确实存在）
2. 若没有必要，不应混用不同目录口径。
3. 优先保证当前仓库路径示例可直接复制运行。
4. 对每条关键命令补充“在哪个目录执行”的说明。

#### 验收标准

- 新用户在项目根目录中可直接照抄主流程命令。
- 不再出现目录结构与当前仓库明显不匹配的主命令示例。

### 5.2 “无需额外安装 / 内置开箱即用”表述过强，容易与真实环境冲突

#### 问题表现

`SKILL.md` 中存在如下表达：

- “本技能已将所有依赖打包到包内，无需额外安装。”
- `openpyxl`、`akshare`、`pandas`、`requests`、`Node.js`、`prosearch.cjs` 均标记为“内置”。

但从当前仓库看：

- 存在 `scripts/requirements.txt`
- 存在 `check_env.py`
- 当前项目本质上仍然依赖外部 Python/Node 运行环境

这说明“完全无需安装”的说法，至少在源码仓库运行场景下并不稳妥。

#### 风险

- 执行者预期错误，以为一定能直接运行。
- 一旦本地缺少依赖，第一反应会变成“代码坏了”，而不是“环境未满足”。
- 文档会削弱排障效率。

#### 完善要求

1. 将环境说明改为条件化表述，例如：
   - 若运行环境已预装依赖，则可直接执行。
   - 若 `check_env.py` 报缺失，则按 `scripts/requirements.txt` 补装依赖。
2. 区分：
   - Python 运行时依赖
   - Node 运行时依赖
   - 仓库内自带脚本文件
3. 明确“脚本文件已包含”和“运行依赖已安装”是两件不同的事。

#### 验收标准

- 环境说明不再对所有场景做绝对承诺。
- 文档中的环境描述能和 `check_env.py` 的存在逻辑自洽。

### 5.3 脚本速查虽完整，但缺少“稳定运行优先顺序”

#### 问题表现

当前 `SKILL.md` 有脚本速查表，也有快速开始，但没有把“为了稳定成功率，推荐先后顺序是什么”明确写成一套最小执行闭环。

例如，对于一个新任务，真正稳定的顺序通常应更像：

1. 先确认环境
2. 再确认年报是否已发布
3. 再用 AKShare 诊断脚本探测接口和字段
4. 再跑一键提取或主流程
5. 若失败，再进入 ProSearch / 手工补齐 / 模板填充
6. 最后做验证和回读

当前文档虽已包含这些内容，但它们分散在不同章节，执行顺序仍需读者自己拼出来。

#### 风险

- 新执行者可能直接跑主流程，跳过环境检查和发布状态检查。
- 出问题后才去补看前置步骤，导致排障成本高。
- 文档详细但仍不够“可落地”。

#### 完善要求

新增一节：`稳定运行最小闭环（推荐执行顺序）`，明确写成固定顺序，例如：

### 推荐顺序
1. `python scripts/check_env.py`
2. 搜索确认年报是否正式发布
3. `python scripts/akshare_fetch.py ...` 做接口探测
4. `python scripts/extract_data.py ...` 跑主链路
5. 必要时执行手动补齐
6. 生成表2
7. 执行验证和回读
8. 检查文件是否真实落盘

#### 验收标准

- 不看全文，只看该章节也能完成一次相对稳妥的执行。
- 主流程前置动作和异常分流动作顺序清楚。

### 5.4 年报发布状态检查是关键规则，但应进一步工程化

#### 问题表现

当前文档已经明确提出：

- 必须先检查年报是否已发布
- 未发布时不能直接说“年报显示”
- 应区分年报、业绩快报、业绩预告、季报

这条规则很对，也非常关键。但目前仍偏“规则说明”，还不够工程化。

#### 风险

- 执行者知道要查，但不知道查完后如何将结果写入后续链路。
- 缺少固定的状态输出模板，容易在后续步骤中又混淆口径。
- 表1、表2、最终回复可能出现不同口径。

#### 完善要求

1. 把“发布状态检查”升级为一个固定输入项。
2. 明确检查结果至少要输出：
   - 状态：已发布 / 未发布 / 仅快报 / 仅预告 / 仅季报
   - 截止日期
   - 使用口径
   - 文本表述模板
3. 给出标准写法示例，例如：

```text
发布状态：未发布
当前使用口径：2025年业绩快报
标准表述：业绩快报显示……，年报尚未正式发布
```

4. 要求后续所有步骤沿用同一口径，不允许在中途切换成“年报显示”。

#### 验收标准

- 发布状态检查不仅是“先搜一下”，而是变成可复用的标准化前置结果。
- 表述口径在主流程、Excel、分析结论中一致。

### 5.5 五级信息优先级规则很强，但缺少“稳定执行版”说明

#### 问题表现

当前文档要求全部字段严格按五级优先级逐级查询，并且“哪一级先查到有效数据就直接使用”。

这条规则在质量控制上是合理的，但在实际高频执行中，存在两个问题：

1. 规则虽然正确，但实际操作成本较高。
2. 没有明确区分“全量严格执行”和“在时间受限时的最低可交付执行法”。

#### 风险

- 执行者可能形式化理解，机械穷举所有字段所有层级，导致效率过低。
- 或者反过来，由于执行成本过高，实际又跳过规则，形成“写在文档里但跑不出来”的情况。

#### 完善要求

在不削弱五级优先级原则的前提下，新增两套说明：

### 标准执行法
- 适用于正式交付任务
- 关键字段尽量完整按五级顺序检索
- 对每项写明来源级别

### 稳定降级执行法
- 当时间、网络、接口可用性不足时
- 核心字段必须优先保住：营收、归母净利润、扣非净利润、毛利率、净利率、经营现金流、资产负债率、ROE、市值、主营业务
- 非核心字段无法获取时允许标注“数据缺口”，但必须说明已查到哪一级、为何中止

#### 验收标准

- 五级优先级规则仍保留为主规则。
- 同时新增“执行落地版本”，提升可操作性和真实完成率。

### 5.6 AKShare 部分说明较多，但诊断闭环仍可进一步强化

#### 问题表现

当前文档已经提供：

- AKShare 全市场接口说明
- 故障时先跑 `akshare_fetch.py`
- 常见错误类型和切换策略

这已经很好，但要达到“稳定运行手册”标准，还缺少更强的排障闭环。

#### 当前不足

1. 缺少“先确认什么，再确认什么”的固定排查顺序。
2. 缺少对 `akshare_fetch.py` 输出结果的判读规则。
3. 缺少“什么情况下改映射，什么情况下不要改主流程”的明确信号。
4. 缺少“字段变更”和“接口不可用”的区别处理标准。

#### 完善要求

新增 `AKShare 标准排障闭环` 章节，至少包括：

### 第一步：判断故障类型
- 环境缺失
- 网络超时/代理问题
- 接口空数据
- 字段名变化
- symbol 格式错误
- 市场识别错误

### 第二步：固定排查顺序
1. 跑 `check_env.py`
2. 跑 `akshare_fetch.py`
3. 看是否有数据返回
4. 看字段是否变更
5. 看股票代码和市场识别是否正确
6. 仅在诊断确认后，才改 `akshare_universal_integration.py`

### 第三步：输出判读模板
例如：

```text
诊断结果：接口可访问，但字段名变化
建议动作：仅更新字段映射，不修改主流程入口
```

#### 验收标准

- 执行者能按文档从“报错”快速收敛到“具体该改哪里”。
- 文档能减少盲目修改主流程的概率。

### 5.7 ProSearch / 全网搜索部分强调很多，但缺少“投入上限”说明

#### 问题表现

当前文档对全网搜索修复写得很细，包含：

- 多轮搜索建议
- 多个关键词模板
- 不同类型数据的最佳免费来源
- 修复目标

这对补齐质量很有帮助，但如果目标是稳定生成效果，还需要明确“搜索增强的投入边界”。

#### 风险

- 执行者可能在边缘字段上耗费过多时间。
- 主任务从“稳定生成双表”演变成“无限制补数”。
- 文档会鼓励高投入，但未明确何时应停止并进入交付。

#### 完善要求

在保留搜索增强策略的基础上，新增：

### 搜索增强停止条件
- 已满足表1核心字段完整度要求
- 无法在限定轮次内获取更高质量来源
- 新增搜索结果质量低于当前结果
- 剩余字段属于非核心字段，且已能明确标注缺口原因

### 建议执行上限
- 默认先做 1-2 轮关键字段修复
- 仅在正式深度任务时再扩展到 4-6 轮

#### 验收标准

- 文档既能指导补数，也能防止无限扩张。
- 搜索部分更服务于“稳定交付”，而不是单纯追求信息越多越好。

  - 哪些是必须掌握的主链路示例

#### 完善要求

对示例代码进行分层标记：

### A类：主链路必读示例

### 5.9 双表“必须生成”目标明确，但极端失败场景的补救手册还不够具体

#### 问题表现

当前文档强烈强调：

- 只要触发技能，最终必须生成表1和表2
- 不得只给文字汇总
- 任一文件未生成不可算完成

这些目标是对的，但若要真正提升稳定性，还需要把“无法直接成功时怎么办”写得更具体。

#### 当前不足

极端失败场景虽有原则性说明，但仍缺少细化补救路径，例如：

- AKShare 失败 + ProSearch 失败 + 仍需交付时，最小补救步骤是什么？
- 表1已生成但表2为空时，如何快速补生成？
- 验证报告写入失败时，是整体失败还是允许单项补救？
- Excel 文件已生成但无法读取时，先检查什么？

#### 完善要求

新增 `失败补救运行手册` 章节，按场景给出固定动作：

### 场景A：AKShare失败
- 先跑诊断脚本
- 再改字段映射或切搜索
- 核心字段保底从官方披露或已有资料手工填入

### 场景B：表1生成失败
- 先生成空模板
- 再填充可确认字段
- 缺失字段标注缺口
- 至少完成最小可交付版表1

### 场景C：表2生成失败
- 基于表1关键字段手工生成简版分析表
- 保证6维度结构存在
- 内容不足时明确写“信息不足”而不是留空

### 场景D：文件生成但不可读
- 立即执行回读
- 若回读失败，重生成
- 必要时更换输出文件名重试

#### 验收标准

- 任一关键环节失败时，都有可执行的补救动作，而不是只给原则。
- 文档能把“失败”导向“补救交付”，而不是直接中断。

### 5.10 数据验证章节存在，但缺少“验证失败后的处理顺序”

#### 问题表现

当前文档写了：

- 五维度验证
- 异常值范围
- 交叉验证容忍度
- 时效性检查
- 来源可靠性检查

但对执行者而言，更关键的问题往往是：

> 验证失败后先修什么、后修什么？

#### 风险

- 执行者看到了很多验证规则，但不知道如何根据优先级修复。
- 容易在低优先级问题上耗时，耽误主任务交付。

#### 完善要求

补充 `验证失败处理顺序`：

### P0：必须先修
- 文件不可读
- 表1/表2缺失
- 核心字段为空
- 来源未标注
- 年报口径表述错误

### P1：建议再修
- 关键财务数据交叉验证差异过大
- 明显异常值
- 时效性超限

### P2：可视时间再修
- 非核心字段缺口
- 表述细节优化
- 更多辅助来源补充

#### 验收标准

- 验证章节不仅能指出问题，还能指挥修复顺序。
- 文档更贴近真实交付场景。

---

### 5.11 输出文件要求完整，但“落盘确认”和“可读确认”还应更硬化

#### 问题表现

当前文档明确要求：

- 两份 Excel 必须真实落盘
- 最终回复需列出完整路径
- 文件必须可读

这些要求很好，但如果目标是“稳定生成效果”，还需要把“如何确认”写成固定动作，而不是只写结果要求。

#### 完善要求

新增 `输出后固定检查动作`：

1. 检查表1文件是否存在
2. 检查表2文件是否存在
3. 用 `read_excel.py` 或 pandas 回读
4. 检查 Sheet 是否齐全
5. 检查表1是否为22项结构
6. 检查表2是否为6维度结构
7. 检查路径是否为最终交付路径

并给出示例命令：

```bash
python scripts/read_excel.py "输出文件.xlsx" --n 5
```

#### 验收标准

- 交付前有固定回读动作。
- “生成成功”不再只凭脚本打印，而要以实际文件可读为准。

### 5.12 维护原则存在，但还应加强“文档与脚本同步更新”约束

#### 问题表现

当前 `SKILL.md` 已经写了最小修改原则、隔离性、更新规范等。这对代码维护有帮助。

但从运行手册角度，还需要补一条重要原则：

> 主脚本改了，`SKILL.md` 中的命令、参数、排障说明、示例代码也必须同步更新。

#### 风险

- 代码已变，文档仍是旧参数和旧命令。
- 文档越详细，过时后误导性越强。

#### 完善要求

新增文档同步约束：

1. 修改脚本 CLI 参数后，必须检查 `SKILL.md` 中对应示例。
2. 修改主流程后，必须检查“快速开始”“标准流程”“失败补救”章节。
3. 修改验证逻辑后，必须检查“数据验证”和“验收标准”章节。
4. 新增脚本后，必须决定是否加入“脚本速查”和“推荐执行顺序”。

#### 验收标准

- `SKILL.md` 不再只是静态说明，而成为和脚本同频维护的运行文档。

## 10. 最终结论

如果把 `SKILL.md` 定位为“运行手册型技能文档”，那么它当前最大的问题并不是“太长”或“细节太多”，而是：

- **部分细节还不够贴近当前项目实际**
- **部分强规则还不够工程化和可降级**
- **部分排障思路还没有闭环化**
- **部分示例还没有围绕稳定跑通主链路进行分层组织**

因此，后续完善方向不应是简单删减内容，而应是：

1. 保留高密度手册特征
2. 提升命令和路径真实性
3. 提升排障闭环能力
4. 提升失败补救可执行性
5. 提升双表稳定交付成功率
6. 提升文档与脚本的长期同步程度

也就是说，`SKILL.md` 未来应当继续做“重型操作手册”，但要从“内容很多”升级为“内容很多且真的能稳定指导执行”。
