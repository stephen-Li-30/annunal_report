# 年报数据准确性保障机制

## 概述

本文档详细说明如何提升 annual-report-extractor 技能的数据准确性，确保提取的年报数据可靠、可信、可验证。

核心原则：**项目不再根据字段特性选择最终最合适来源，而是统一按五级顺序查询，哪一级先查询到有效数据，就使用哪一级，并严格标注来源级别。**

---

## 一、五维度数据准确性验证体系

### 1.1 验证维度概览

| 维度 | 权重 | 验证内容 | 通过标准 |
|------|------|---------|---------|
| **完整性** | 20% | 22项数据是否齐全 | 无缺失项（100分） |
| **合理性** | 25% | 异常值检测 | 无ERROR级异常 |
| **一致性** | 25% | AKShare交叉验证 | 差异在容忍范围内 |
| **时效性** | 15% | 数据是否为最新财报 | 财务数据≤1年，市值数据≤30天 |
| **可靠性** | 15% | 来源级别标注与规则一致性 | 核心数据有来源级别标注，来源不明项为0 |

### 1.2 总体得分计算

```
总体得分 = 完整性得分×0.2 + 合理性得分×0.25 + 一致性得分×0.25 + 时效性得分×0.15 + 可靠性得分×0.15
```

**评分标准：**
- 90-100分：优秀，数据质量高
- 70-89分：良好，有少量警告
- 50-69分：及格，有错误需修复
- <50分：不合格，需要重新提取

## 二、异常值检测规则

### 2.1 财务指标合理范围

| 指标 | 最小值 | 最大值 | 单位 | 说明 |
|------|--------|--------|------|------|
| 毛利率 | -20 | 95 | % | 大多数行业在-10%~90%之间 |
| 净利率 | -50 | 60 | % | 超过范围需验证 |
| ROE | -100 | 200 | % | 极端值需验证 |
| 负债率 | 0 | 100 | % | 超过范围需验证 |
| 营收增长率 | -100 | 500 | % | 极端增长需验证 |
| 净利润增长率 | -1000 | 1000 | % | 极端值需验证 |
| 每股收益 | -10 | 100 | 元 | 极端值需验证 |
| 每股净资产 | 0 | 1000 | 元 | 极端值需验证 |
| 存货周转天数 | 0 | 1000 | 天 | 超过需验证 |
| 应收账款周转天数 | 0 | 500 | 天 | 超过需关注 |
| 总资产周转率 | 0 | 10 | 次 | 超过需验证 |
| 流动比率 | 0 | 20 | 倍 | 超过可能异常 |
| 速动比率 | 0 | 20 | 倍 | 超过可能异常 |

### 2.2 必须为正数的指标

| 指标 | 说明 | 异常处理 |
|------|------|---------|
| 总资产 | 公司总资产 | 负数为ERROR |
| 净资产 | 股东权益 | 负数为ERROR |
| 注册资本 | 公司注册资本 | 负数为ERROR |
| 员工人数 | 员工总数 | 负数为ERROR |
| 门店数量 | 门店总数 | 负数为ERROR |
| 总股本 | 股份总数 | 负数为ERROR |
| 营收 | 营业收入 | 负数为ERROR |
| 净利润 | 净利润（亏损为负） | 需根据情况判断 |

### 2.3 异常值处理流程

```
检测到异常值
    ↓
判断严重程度
    ├── ERROR（严重）：数据明显错误
    │       └── 标记为ERROR，阻止生成Excel
    │       └── 必须修复后才能继续
    │
    └── WARNING（警告）：数据可能异常
            └── 标记为WARNING
            └── 建议核查，但可以继续
            └── 如果用户确认，手动覆盖
```

### 2.4 常见异常值案例

| 异常类型 | 典型案例 | 原因分析 | 处理方法 |
|---------|---------|---------|---------|
| 毛利率>95% | 98% | 软件/特许经营行业可能是真的 | 核查行业特性 |
| 毛利率<0 | -10% | 主营业务亏损 | 确认数据来源 |
| ROE>100% | 150% | 高杠杆公司可能 | 核查负债结构 |
| ROE<0 | -50% | 公司亏损 | 确认是亏损数据 |
| 营收增长>100% | 500% | 并购/新业务 | 核查是否有重大事件 |
| 营收增长<-50% | -80% | 主业萎缩 | 确认数据年份 |

## 三、多源交叉验证规则

### 3.1 AKShare数据对比

**核心财务数据容差标准：**

| 指标 | 容差 | 说明 |
|------|------|------|
| 营收 | 1% | 极其精确 |
| 净利润 | 5% | 可能有口径差异 |
| 毛利率 | 2% | 计算方式可能不同 |
| ROE | 5% | 计算方式可能不同 |
| 负债率 | 2% | 数据来源一致 |

**交叉验证流程：**

```
Step 1: 获取用户提取的数据
Step 2: 调用AKShare获取同公司数据
Step 3: 逐项对比核心指标
Step 4: 计算差异百分比
Step 5: 判断是否在容差范围内
        ├── 在容差内 → 标记为"已交叉验证"
        └── 超出容差 → 标记为"不一致"，需核查
```

### 3.2 内部逻辑一致性检查

**ROE验证公式：**

```
ROE = 净利润 / 净资产 × 100%
```

如果报告的ROE与计算值差异>10%，需要核查。

**毛利率验证公式：**

```
毛利率 = (营收 - 营业成本) / 营收 × 100%
```

如果报告的毛利率与计算值差异>5%，需要核查。

**负债率验证公式：**

```
负债率 = 总负债 / 总资产 × 100%
```

如果报告的负债率与计算值差异>5%，需要核查。

### 3.3 时间序列一致性检查

**营收/净利润趋势检查：**

```
如果 2025年营收 > 2024年营收 > 2023年营收（持续增长）
但 2025年净利润 < 2024年净利润 < 2023年净利润（持续下降）
→ 标记为WARNING，需核查
→ 可能原因：成本上升、费用增加、资产减值等
```

## 四、数据时效性保障

### 4.1 数据时效性标准

| 数据类型 | 最大时效 | 超过处理 |
|---------|---------|---------|
| 财务数据（年报） | 1个财年 | 标记WARNING，提示更新 |
| 财务数据（季报） | 1个季度 | 标记WARNING，提示更新 |
| 市值数据 | 30天 | 标记WARNING，提示更新 |
| 估值数据（PE/PB） | 30天 | 标记WARNING，提示更新 |
| 股东数据 | 90天 | 标记WARNING，提示更新 |

### 4.2 年报数据时效性判断

**最新年报判断逻辑：**

```python
current_year = datetime.now().year
data_year = int(data["财务年度"])

if data_year < current_year - 2:
    # 数据过旧，ERROR
    issues.append({
        "issue": "数据年份过旧",
        "severity": "ERROR"
    })
elif data_year < current_year:
    # 不是最新，WARNING
    issues.append({
        "issue": "数据不是最新年份",
        "severity": "WARNING"
    })
```

### 4.3 市值数据时效性判断

```python
data_date = datetime.strptime(data["数据日期"], "%Y-%m-%d")
days_old = (datetime.now() - data_date).days

if days_old > 30:
    issues.append({
        "issue": "市值数据超过30天",
        "severity": "WARNING",
        "days_old": days_old
    })
```

## 五、来源级别与规则一致性验证

### 5.1 来源标注要求

| 数据类型 | 必须标注 | 标注内容 |
|---------|---------|---------|
| 全部字段 | ✅ | 来源级别 + 具体来源 |
| 市值/估值/实时数据 | ✅ | 来源级别 + 数据日期 |
| 第5级数据 | ✅ | 来源级别 + 估算/预测/推测说明 |
| 数据缺口 | ✅ | 数据缺口 + 原因说明 |

### 5.2 来源级别说明

| 级别 | 标注 | 数据来源 | 示例 |
|------|------|---------|------|
| 第1级 | 官方/监管披露 | 年报/10-K/SEC EDGAR/HKEX/巨潮/公司IR | "来源级别:第1级 | 来源:公司年报2025" |
| 第2级 | 权威财经数据库/接口 | AKShare/东方财富/同花顺/新浪/Wind等 | "来源级别:第2级 | 来源:AKShare实时行情接口" |
| 第3级 | 行业机构/研究报告 | IDC/Omdia/Gartner/券商研报等 | "来源级别:第3级 | 来源:IDC 2025行业报告" |
| 第4级 | 财经媒体/全网搜索 | 证券之星/中证智能财讯/财联社/ProSearch等 | "来源级别:第4级 | 来源:ProSearch搜索结果" |
| 第5级 | 估算/预测/人工分析 | 可比公司测算/分析师预测/人工推断 | "来源级别:第5级 | 来源:可比公司样本测算" |
| 缺口 | 数据缺口 | 无法获取 | "来源级别:未获取 | 可靠性:数据缺口" |

### 5.3 来源检查流程

```python
critical_items = [
    "营收", "净利润", "毛利率", "ROE", "负债率", "市值"
]

for item in critical_items:
    source = data.get(f"{item}_来源", "")
    
    if not source:
        # 缺少来源，ERROR
        pass
    elif "来源级别" not in source:
        # 缺少来源级别，WARNING
        pass
    elif any(level in source for level in ["第1级", "第2级", "第3级", "第4级", "第5级"]):
        # 合规来源级别
        pass
    elif "数据缺口" in source:
        # 合规缺口标注
        pass
    else:
        # 来源不明，WARNING
        pass
```

验证重点不是要求字段必须来自某个固定来源，而是检查：是否按顺序查询、是否首次命中即采用、是否正确标注来源级别。
        issues.append({
            "item": item,
            "issue": f"核心数据使用了不可靠来源：{reliability}",
            "severity": "WARNING"
        })
```

## 六、数据修复流程

### 6.1 验证失败处理流程

```
验证结果：ERROR
    ↓
判断错误类型
    ├── 完整性问题
    │       └── 执行全网搜索修复流程（Step 4.5）
    │
    ├── 合理性异常
    │       ├── 核查数据来源
    │       ├── 重新计算数据
    │       └── 如果确认无误，标注"已人工确认"
    │
    ├── 一致性问题
    │       ├── 以AKShare数据为准
    │       └── 或使用最新官方年报数据
    │
    ├── 时效性问题
    │       └── 获取最新数据
    │
    └── 来源可靠性问题
            └── 补充来源标注
            └── 或升级数据来源
```

### 6.2 全网搜索修复流程

当存在"行业估算"或"数据缺口"项时，执行6轮ProSearch搜索：

```
Phase 1: 识别需要修复的数据项
Phase 2: 分批并行搜索（每轮20条，至少4-6轮）
    ├── 轮1：PE百分位 + PB百分位
    ├── 轮2：供应商 + 客户
    ├── 轮3：股东增减持
    ├── 轮4：合同负债 + 资本开支
    ├── 轮5：行业平均数据
    └── 轮6：市场份额 + 美股同类公司
Phase 3: 抓取关键页面深度解析
Phase 4: 数据分级入库
```

**目标：将"行业估算"项从8-10个降至2-3个**

### 6.3 数据修复效果评估

| 指标 | 初始 | 修复后 | 目标 |
|------|------|--------|------|
| 行业估算项数量 | 8-10项 | 2-3项 | <3项 |
| 数据缺口数量 | 2-4项 | 0-1项 | <1项 |
| 官方数据比例 | <50% | >80% | >80% |
| 来源标注完整性 | <70% | >95% | >95% |

## 九、总结

### 9.1 五维度验证体系

| 维度 | 目标 | 通过标准 |
|------|------|---------|
| 完整性 | 22项数据齐全 | 无缺失 |
| 合理性 | 无异常值 | 无ERROR |
| 一致性 | 多源一致 | 差异在容差内 |
| 时效性 | 最新数据 | 财务≤1年，市值≤30天 |
| 可靠性 | 来源明确 | 核心数据有标注 |

### 9.2 关键成功因素

1. **AKShare数据优先**：核心财务数据使用AKShare官方数据
2. **交叉验证**：核心数据必须有多源验证
3. **异常值检测**：自动检测明显错误
4. **来源标注**：每项数据必须标注来源和可靠性
5. **持续修复**：使用全网搜索修复行业估算和数据缺口
