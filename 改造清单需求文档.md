# 改造清单需求文档

## 1. 文档目的

本文档用于明确 `annual-report-extractor` 项目中 AKShare 能力的改造目标、范围、优先级、实施原则与验收标准，指导后续以最小修改方式完成项目优化。

本文档聚焦以下问题：
- AKShare 接口使用分散，市场识别与代码标准化不统一
- 字段映射存在硬编码，接口字段变化时易失效
- 错误处理粒度不足，难以快速判断失败原因
- 缓存策略较粗，财报数据与实时行情未分层处理
- 诊断脚本、适配层、主流程职责边界尚不够清晰

---

## 2. 背景说明

当前项目已经具备 AKShare 集成能力，主要涉及以下文件：

- `scripts/extract_data.py`
- `scripts/akshare_universal_integration.py`
- `scripts/akshare_integration.py`
- `scripts/akshare_fetch.py`
- `references/akshare-a-stock-guide.md`
- `references/akshare-universal-guide.md`
- `SKILL.md`

现状特点：

1. 主流程已实现“AKShare 优先，搜索补充”的基本路线。
2. 已有统一集成层，但仍存在接口参数、字段映射、错误处理不统一的问题。
3. 已有 Python 诊断脚本，但能力仍偏基础，尚未形成完整排障规范。
4. 文档已经描述了较完整的 AKShare 能力，但与实际代码仍存在收敛空间。

因此，需要形成分阶段、可落地的改造清单，避免直接在主流程中堆叠临时修复逻辑。

---

## 3. 改造目标

### 3.1 总体目标

将 AKShare 相关能力从“可用但脆弱”升级为“可维护、可诊断、可扩展”的工程化方案。

### 3.2 具体目标

1. 统一股票市场识别与代码标准化逻辑。
2. 统一不同 AKShare 接口的 symbol 输入规则。
3. 建立字段候选映射机制，降低接口字段变化带来的维护成本。
4. 将错误处理从字符串提示升级为结构化错误分类。
5. 对财报数据与实时行情建立分层缓存策略。
6. 明确诊断脚本、适配层、主流程的职责边界。
7. 建立 AKShare 专项验证链路，降低版本升级或接口变更风险。

---

## 4. 改造原则

本次改造必须严格遵循项目已有维护规则。

### 4.1 最小修改原则

- 不重构与本次目标无关的代码。
- 不修改现有脚本的对外接口，除非新增参数为向后兼容方式。
- 尽量通过新增函数、独立模块、局部替换实现改造。

### 4.2 隔离原则

- 主流程、适配层、诊断层分离。
- 不在 `extract_data.py` 中继续堆叠底层 AKShare 调试细节。
- 尽量把易变逻辑收敛到独立函数中。

### 4.3 向后兼容原则

- 保持当前一键提取链路可继续运行。
- 保持现有脚本调用方式基本不变。
- 优先新增能力，不破坏已有功能。

### 4.4 可诊断原则

- 所有 AKShare 失败场景都应尽量可分类、可定位、可复现。
- 当主流程获取失败时，应能够明确提示用户使用独立 Python 诊断脚本。

## 6. 当前问题清单

### 6.1 市场识别逻辑重复且口径不统一

问题表现：
- `extract_data.py` 与 `akshare_universal_integration.py` 中均存在市场识别逻辑。
- 对 `.SH/.SZ/.HK/.HKG`、5位港股代码、含字母美股代码等处理规则不完全一致。

风险：
- 同一股票代码在不同模块中可能被识别为不同市场。
- 主流程与适配层可能出现参数不一致。

### 6.2 不同接口的 symbol 格式规则未统一封装

问题表现：
- 某些接口使用纯代码。
- 某些接口可能需要 `SH600660` 这类前缀。
- 同类接口在不同文件中的调用写法不一致。

风险：
- AKShare 升级后更容易出现局部接口失效。
- 问题难以统一排查。

### 6.3 字段映射硬编码较多

问题表现：
- 大量使用 `row.get("字段名")` 的单字段读取方式。
- 对字段别名、历史字段、不同市场字段差异的兼容不足。

风险：
- 当 AKShare 字段名调整时，主流程会 silently degrade 或写入空值。

### 6.4 错误处理过于粗粒度

问题表现：
- 当前很多逻辑仅记录 `str(e)`。
- 未区分超时、空数据、参数错误、字段缺失、网络错误等类型。

风险：
- 上层无法做针对性重试、降级或提示。
- 调试效率低。

### 6.5 主流程中的 AKShare 处理职责偏重

问题表现：
- `apply_akshare_data` 同时承担缓存、网络请求、数据映射、文本拼装、22项写入等多项职责。

风险：
- 后续维护成本高。
- 出问题时难定位是“取数层问题”还是“填表层问题”。

### 6.6 诊断脚本能力尚不完整

问题表现：
- `akshare_fetch.py` 目前更偏基础探测器。
- 对市场、数据集选择、保存结果、查看字段等支持有限。

风险：
- AKShare 出问题时仍然需要改代码排查，缺少标准排障动作。

### 6.8 缺少 AKShare 专项回归验证

问题表现：
- 现有 `smoke_test.py` 主要覆盖 Excel 生成链路。
- 尚无专门验证 AKShare 适配稳定性的独立测试。

风险：
- AKShare 版本变化时，问题可能在发布后才暴露。

## 8. P0 需求（高优先级，建议优先实施）

### P0-1 统一市场识别与代码标准化

#### 需求描述
新增统一函数，负责：
- 市场识别
- 原始代码解析
- 标准代码输出
- 不同市场的展示代码与接口代码生成

#### 建议产出
新增公共函数，例如：
- `normalize_stock_code(raw_code)`
- 返回结构包含：
  - `market`
  - `normalized_code`
  - `display_code`
  - `raw_code`

#### 影响文件
- `scripts/extract_data.py`
- `scripts/akshare_universal_integration.py`
- 如有必要，可新增独立工具模块

#### 验收标准
以下输入均可正确识别并标准化：
- `600660`
- `600660.SH`
- `000001.SZ`
- `00700`
- `00700.HK`
- `NVDA`
- `AAPL`

---

### P0-2 统一各接口 symbol 适配规则

#### 需求描述
为 AKShare 常用接口建立明确的 symbol 生成规则，不允许在不同文件中重复猜测格式。

#### 覆盖接口建议
- `stock_individual_info_em`
- `stock_financial_abstract`
- `stock_balance_sheet_by_report_em`
- `stock_profit_sheet_by_report_em`
- `stock_cash_flow_sheet_by_report_em`
- `stock_zygc_em`
- 港股财报接口
- 美股财报接口
- 实时行情接口

#### 验收标准
- 同一股票在不同入口调用相同接口时，symbol 规则一致。
- A股主营业务、港股公司信息、美股财务指标等接口至少各验证 1 个样本可用。

---

### P0-3 建立字段候选映射机制

#### 需求描述
将单字段硬编码读取方式改为候选字段映射读取方式。

#### 建议能力
新增类似函数：
- `pick_first_available(row, ["营业收入", "营业总收入", "OPERATE_INCOME"])`
- `pick_column(df, ["REPORT_DATE", "报告期"])`

#### 适用场景
- A股财务分析指标
- 港股分析指标
- 美股分析指标
- 资产负债表字段读取
- 实时行情字段读取

#### 验收标准
- 字段名轻微变化时，不需要改主流程即可正常取值。
- 对关键指标至少覆盖：
  - 报告期
  - 营收
  - 净利润
  - 毛利率
  - ROE
  - 资产负债率
  - 总资产
  - 净资产
  - 合同负债
  - 市值
  - PE
  - PB

---

### P0-4 建立结构化错误分类

#### 需求描述
为 AKShare 取数失败定义标准错误类型。

#### 建议错误类型
- `network_error`
- `timeout_error`
- `empty_data`
- `schema_error`
- `symbol_error`
- `rate_limit_error`
- `unexpected_error`

#### 建议输出结构
```json
{
  "error_type": "schema_error",
  "error_message": "missing column REPORT_DATE",
  "interface": "stock_financial_hk_report_em"
}
```

#### 验收标准
- 主流程能识别错误类型。
- 当出现 schema 类错误时，控制台明确提示优先运行 `akshare_fetch.py` 排查。

## 9. P1 需求（中优先级，建议第二阶段实施）

### P1-1 拆分主流程中的 AKShare 处理职责

#### 需求描述
将 `apply_akshare_data` 中的职责拆分为多个小函数。

#### 建议拆分方向
- `fetch_akshare_payload(...)`
- `map_akshare_payload_to_fields(...)`
- `apply_akshare_fields_to_items(...)`

#### 验收标准
- 单个函数职责更清晰。
- 不改变外部 CLI 用法。
- 保持现有输出结果基本兼容。

---

### P1-2 升级 `akshare_fetch.py` 为标准诊断器

#### 需求描述
将当前脚本增强为正式诊断工具。

#### 建议新增参数
- `--market`
- `--datasets`
- `--spot`
- `--save-json`
- `--show-columns`
- `--limit`

#### 能力目标
- 支持 A股、港股、美股
- 支持只探测指定数据集
- 支持保存诊断结果为 JSON 文件
- 支持展示字段列表而非只展示数据预览

#### 验收标准
能够执行类似命令：
```bash
python scripts/akshare_fetch.py --market 港股 --code 00700 --year 2025 --datasets profile,indicator,balance --format json
```

---

### P1-3 在主流程中增加标准排障提示

#### 需求描述
当 AKShare 获取失败或返回空值时，主流程输出统一提示。

#### 提示内容建议
- 当前失败的市场与代码
- 失败接口或错误类型
- 推荐执行的诊断命令

#### 验收标准
失败后，用户无需读源码即可知道下一步排障动作。

## 13. 验收标准

### 13.1 功能验收

1. A股、港股、美股至少各 1 个样本能够成功走通 AKShare 主链路。
2. 主流程 AKShare 失败时，能给出明确诊断建议。
3. `akshare_fetch.py` 可独立用于排障。
4. 字段名轻微变化时，系统具备一定兼容能力。

### 13.2 工程验收

1. 不破坏现有 CLI 使用方式。
2. 不影响 Excel 双表输出主链路。
3. 新增逻辑有清晰边界，不污染无关模块。
4. 关键脚本改动后可通过相应测试或烟测。

### 13.3 文档验收

1. `SKILL.md` 已同步更新使用规范。
2. AKShare 参考文档与代码实现基本一致。
3. 诊断脚本在文档中有明确入口说明。

---

## 14. 风险与注意事项

1. AKShare 外部接口存在不稳定性，改造目标应以“增强容错与诊断能力”为主，而非追求绝对稳定。
2. 不建议在本次改造中同步大规模清理所有历史脚本，避免引入额外风险。
3. 若 AKShare 某些接口存在版本差异，应优先通过字段兼容与诊断能力吸收变化。
4. 对行情类接口的优化需兼顾性能与接口稳定性，避免单纯追求更快而引入更多失败场景。

## 17. 总结

本次 AKShare 改造的核心不是增加更多接口，而是提升以下三方面能力：

1. 标准化：统一市场、代码、字段口径
2. 可维护：降低接口变化带来的修改成本
3. 可诊断：让 AKShare 出问题时能够快速判断、快速修复

建议后续严格遵循“先诊断脚本、再适配层、最后主流程”的改造顺序，避免继续在主流程中堆叠临时修复逻辑。
