# 脚本修复开发计划

## 1. 文档目的

本文档用于根据《脚本修复需求文档》制定一份可执行、可验证、低风险的修复开发计划。

本计划的核心目标不是扩展业务功能，而是：

1. 修复脚本帮助文案、提示信息与 `SKILL.md` 之间的不一致
2. 统一主链路命令口径，默认全部收敛到“仓库根目录运行方式”
3. 让 `SKILL.md` 的交付标准与 `data_validator.py` 当前更严格的“可继续交付”规则对齐
4. 明确 `smoke_test.py` 的离线烟测边界，避免误读
5. 收敛脚本中不必要的说明性注释，使维护风格更一致

本计划遵循以下总原则：

- 不引入新依赖
- 不扩展新业务能力
- 不放宽 `data_validator.py` 的现有交付门槛
- 优先修正文档、帮助说明和命令口径
- 优先小步修复，避免牵动主流程逻辑

---

## 2. 开发目标

### 2.1 总体目标

使项目中的关键脚本说明、主文档说明、验收口径和烟测定位保持一致，减少执行者在“看文档”和“看脚本输出”之间的理解冲突。

### 2.2 具体目标

1. `check_env.py` 的使用说明、依赖安装提示、下一步提示与 `SKILL.md` 保持一致。
2. `extract_data.py` 顶部帮助文案和失败后的补救命令与 `SKILL.md` 保持一致。
3. `generate_analysis_report.py` 顶部使用示例与真实 CLI 参数要求一致。
4. `SKILL.md` 的交付标准与 `data_validator.py` 的“可继续交付”规则一致。
5. `SKILL.md` 中补充 `smoke_test.py` 的边界说明。
6. 收敛明显不必要的说明性注释，保持脚本维护风格统一。

## 3. 本次修改范围

### 3.1 主要修改对象

- `SKILL.md`
- `scripts/check_env.py`
- `scripts/extract_data.py`
- `scripts/generate_analysis_report.py`

2. 不重构 `extract_data.py` 主流程业务逻辑。

4. 不新增统一流水线脚本。
5. 不新增依赖和复杂自动化机制。

## 4. 修复原则

### 4.1 口径统一原则

- 同一命令只保留一种默认主口径。
- 当前默认主口径统一为：**仓库根目录运行方式**。
- 若保留其他运行方式，也只能作为补充说明，而不是默认示例。

### 4.2 文档服从脚本能力原则

- 对验收门槛，以 `data_validator.py` 当前规则为准。
- 不为迁就旧文档而放宽脚本交付标准。
- 文档应反映真实脚本行为，而不是反过来要求脚本变松。

### 4.3 小范围修复原则

- 优先修正帮助文案、终端提示和主文档说明。
- 只有当说明与脚本行为明显冲突时，才局部调整对应脚本文本。
- 避免借修复说明之机改动主流程业务逻辑。

### 4.4 可验收原则

- 每项修复都应有明确验证方法。
- 修复完成后，应能通过阅读脚本帮助文案和主文档，得到一致结论。

## 5. 分阶段开发计划

## 5.1 第一阶段：统一环境检查脚本口径

### 目标

让 `check_env.py` 的帮助说明、依赖安装提示、下一步建议与 `SKILL.md` 当前主口径一致。

### 任务清单

#### 任务 1：修复 `check_env.py` 顶部使用说明
- 将文件头部默认运行方式统一为仓库根目录口径。
- 默认示例改为：
  - `python scripts/check_env.py`

#### 任务 2：修复 `check_env.py` 依赖安装提示
- 将失败提示中的依赖安装命令统一为：
  - `pip install -r scripts/requirements.txt`

#### 任务 3：修复 `check_env.py` 成功后的下一步提示
- 将环境检查通过后的下一步命令统一为仓库根目录口径。
- 例如：
  - `python scripts/extract_data.py --help`

### 预期产出

- `check_env.py` 文件头部说明与运行提示完全统一。
- 执行者从 `SKILL.md` 或 `check_env.py` 任一入口进入，都不会遇到路径口径冲突。

### 验收方式

- 检查 `check_env.py` 顶部说明是否已改为仓库根目录方式。
- 检查缺依赖提示是否引用 `scripts/requirements.txt`。
- 检查成功后的下一步建议是否引用 `python scripts/extract_data.py --help`。

---

## 5.2 第二阶段：统一主流程脚本帮助文案与补救命令口径

### 目标

让 `extract_data.py` 和 `generate_analysis_report.py` 中的帮助文案、示例命令、失败补救命令与 `SKILL.md` 和真实 CLI 保持一致。

### 任务清单

#### 任务 4：修复 `extract_data.py` 顶部帮助文案
- 统一以下内容为仓库根目录口径：
  1. 主流程运行示例
  2. `--help` 示例
  3. 运行前环境检查命令

#### 任务 5：修复 `extract_data.py` 的失败补救命令
- 将表2生成失败后给出的手动执行提示统一为：
  - `python scripts/generate_analysis_report.py ...`

#### 任务 6：修复 `generate_analysis_report.py` 顶部使用说明
- 将顶部“使用方法”改为完整 CLI 参数版本。
- 至少覆盖：
  - `--company`
  - `--code`
  - `--year`
  - `--input`
  - `--output`
- 与脚本真实 `argparse` 要求保持一致。

### 预期产出

- 关键主流程脚本说明不再保留旧目录口径。
- 用户在失败场景下看到的补救命令也与主文档一致。
- `generate_analysis_report.py` 的示例不再出现“文案可看但无法直接执行”的问题。

### 验收方式

- 检查 `extract_data.py` 顶部文案是否全部使用 `python scripts/...` 形式。
- 检查 `extract_data.py` 的失败补救命令是否已统一。
- 检查 `generate_analysis_report.py` 顶部示例是否与真实 `argparse` 参数一致。

---

## 5.3 第三阶段：统一 `SKILL.md` 验收标准与烟测边界说明

### 目标

让 `SKILL.md` 的交付标准与 `data_validator.py` 当前更严格的“可继续交付”规则对齐，并明确 `smoke_test.py` 只代表离线链路烟测，不代表真实任务交付通过。

### 任务清单

#### 任务 7：升级 `SKILL.md` 的最小验收标准
- 将当前较宽松的交付要求，升级为与 `data_validator.py` 保持一致的门槛。
- 至少补充以下要求：
  1. 表1文件存在
  2. 表1可回读
  3. 表1结构为 22 项
  4. 表2文件存在
  5. 表2可回读
  6. 已执行验证
  7. 数值问题已清零或已解释
  8. 来源标注问题已清零
  9. 来源不明项为 0
  10. 五维度评分达到可继续交付阈值
  11. 以 `data_validator.py` 的“可继续交付”结果作为最终交付门槛

#### 任务 8：补充 `smoke_test.py` 的边界说明
- 在 `SKILL.md` 的诊断入口或烟测说明中明确：
  - `smoke_test.py` 是离线烟测
  - 它验证的是本地生成链路
  - 它不能替代真实公司、真实年份、真实外部数据源下的最终交付验收

### 预期产出

- `SKILL.md` 与 `data_validator.py` 不再存在双重交付标准。
- 烟测通过不会再被误读为真实交付已通过。

### 验收方式

- 检查 `SKILL.md` 的最小验收标准是否已对齐 `data_validator.py`。
- 检查 `SKILL.md` 是否已补充 `smoke_test.py` 的边界说明。
- 检查是否还存在“表1/表2存在即可视为可交付”的宽松表述。

---

## 5.4 第四阶段：收敛说明性注释与维护风格

### 目标

在不影响逻辑的前提下，收敛脚本中不必要的说明性注释，使其与项目当前维护原则一致。

### 任务清单

#### 任务 9：识别可收敛的说明性注释
重点识别以下类型：
- 只是在重复下一行代码含义的注释
- 只是在描述“我现在做什么”的过程性注释
- 与长期维护无关、只服务某次修改过程的说明性注释

#### 任务 10：保留必要注释，删除无效注释
- 保留非显然逻辑、约束、兼容性边界、风险点说明
- 删除口语化、重复性、过程性注释

### 预期产出

- 脚本注释风格更统一
- 注释更偏向长期维护价值，而不是临时说明

### 验收方式

- 抽查 `akshare_fetch.py`、`data_validator.py` 等近期修改脚本
- 检查是否仍残留明显的过程性或重复性注释
- 确认保留下来的注释都能解释非显然逻辑或约束

---

## 6. 推荐实施顺序

建议按以下顺序推进：

1. 先修 `SKILL.md` 的验收标准与 `smoke_test.py` 边界说明
2. 再修 `check_env.py` 的帮助文案与依赖安装提示
3. 再修 `extract_data.py` 的顶部文案与失败补救命令
4. 再修 `generate_analysis_report.py` 的顶部使用说明
5. 最后收敛说明性注释

说明：
- 第 1 步优先，是因为验收标准属于全局口径问题，必须先统一主标准。
- 第 2～4 步属于文本与帮助文案对齐，风险较低，可快速收口。
- 第 5 步属于风格整理，应放在功能和口径一致之后进行。

---

## 7. 风险与应对

### 风险 1：修帮助文案时遗漏某些旧命令口径
**应对：**
- 修复后统一全文搜索 `python check_env.py`、`python extract_data.py`、`python generate_analysis_report.py`
- 对照 `SKILL.md` 脚本速查和主链路章节逐项核对

### 风险 2：`SKILL.md` 口径调整后变得过于复杂
**应对：**
- 只升级交付门槛描述，不重新扩写成长篇验证手册
- 继续坚持“脚本负责判断，文档负责说明入口和标准”

### 风险 3：误把烟测结果写成真实交付证明
**应对：**
- 在 `SKILL.md` 中明确写清 `smoke_test.py` 的适用边界
- 强调最终交付仍以真实表1、表2、回读结果与验证结果为准

### 风险 4：注释清理过程中误删有用说明
**应对：**
- 先识别，再小范围删改
- 优先删除明显口语化、重复性注释
- 对涉及边界和兼容性的注释保守处理

---

## 8. 最终验收标准

完成本开发计划后，应达到以下目标：

### 8.1 命令口径一致
- `SKILL.md`、`check_env.py`、`extract_data.py`、`generate_analysis_report.py` 的默认命令全部统一为仓库根目录运行方式。

### 8.2 CLI 示例真实可复制
- 关键脚本顶部帮助说明与真实参数要求一致。
- 失败提示中的补救命令可直接复制使用。

### 8.3 交付标准一致
- `SKILL.md` 的交付门槛与 `data_validator.py` 的“可继续交付”判定一致。
- 不再出现文档放宽、脚本收紧的双重标准。

### 8.4 烟测定位清楚
- `smoke_test.py` 被明确定义为离线链路烟测。
- 执行者不会把烟测通过误读为真实交付任务通过。

### 8.5 维护风格更统一
- 说明性注释明显收敛。
- 文档与脚本更容易长期同步维护。

---

## 9. 结论

根据《脚本修复需求文档》，本次修复开发的本质不是“改业务”，而是“统一口径、统一标准、统一边界”。

本轮开发完成后，项目应达到以下状态：

1. 执行者从主文档和脚本帮助文案看到的是同一套命令口径。
2. 验收判断以真实脚本能力为准，而不是以较宽松的旧文档为准。
3. 烟测、验收、主流程三者边界清楚，不再混淆。
4. 后续维护更容易通过统一标准持续收口，而不是继续产生新的文档与脚本偏差。
