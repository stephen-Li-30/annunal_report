# 脚本修复需求文档

## 1. 文档目的

本文档用于基于当前 `SKILL.md` 与现有脚本实现之间的差异，形成一份**只描述修复需求、不直接修改脚本代码**的修复说明文档。

本次文档聚焦两类问题来源：

1. 之前检查结果中归类为“**不完全符合的地方**”的问题
2. 之前检查结果中归类为“**存在明显偏差或风险的地方**”的问题

同时，本次需求文档有一个明确约束：

> **不要求放宽 `data_validator.py` 的“可继续交付”规则；相反，应以 `data_validator.py` 当前更严格的规则作为准绳，反向修正文档口径。**

也就是说：

- 不修改 `data_validator.py` 当前“可继续交付”判定逻辑
- 后续如需修复，应优先修改 `SKILL.md`、帮助文案、验收描述和相关说明，使其与 `data_validator.py` 保持一致

---

## 2. 本次需求范围

### 2.1 纳入本次修复需求的问题

本次纳入修复需求的问题包括：

1. `check_env.py` 的命令口径与 `SKILL.md` 不一致
2. `check_env.py` 的安装提示路径与 `SKILL.md` 不一致
3. `extract_data.py` 顶部帮助文案仍保留旧运行口径
4. `extract_data.py` 中手动补救命令仍保留旧运行口径
5. `generate_analysis_report.py` 顶部“使用方法”与真实 CLI 不完全一致
6. `SKILL.md` 的最小验收标准与 `data_validator.py` 的“可继续交付”规则存在层级偏差
7. `SKILL.md` 对 `smoke_test.py` 的定位需要进一步明确边界，避免被误读为“真实交付已通过”

### 2.2 本次不要求改动的问题

以下问题**不在本次文档中要求放宽或撤销**：

#### 不修改项 A：`data_validator.py` 的“可继续交付”规则更严格

该问题不按“脚本太严格，需要放宽”处理。

本次明确采用以下原则：

- `data_validator.py` 当前“可继续交付”规则保留不变
- 后续修复方向应是：
  - 调整 `SKILL.md` 的验收口径
  - 调整相关帮助文案
  - 调整脚本说明文档
  - 让文档与脚本一致
- 不以降低验收门槛的方式来实现“文档一致”

## 3. 当前问题汇总

## 3.1 `check_env.py` 使用说明与仓库根目录口径不一致

### 问题描述

当前 `SKILL.md` 已明确默认推荐：

```bash
python scripts/check_env.py
```

但 `check_env.py` 文件头部的使用方法仍保留为：

```bash
python check_env.py
```

这与当前主文档的统一口径不一致。

### 风险

- 执行者会误以为默认运行方式是在 `scripts/` 目录下
- 与 `SKILL.md` 当前“仓库根目录优先”的要求冲突
- 后续维护时会继续出现文档与脚本帮助说明双口径并存

### 修复要求

后续修复时，应将 `check_env.py` 文件头部使用说明统一为仓库根目录口径，例如：

```bash
python scripts/check_env.py
```

### 验收标准

- `check_env.py` 顶部说明与 `SKILL.md` 保持一致
- 执行者看到脚本帮助文案时，不会与主文档产生路径理解冲突

---

## 3.2 `check_env.py` 安装依赖提示路径不一致

### 问题描述

当前 `SKILL.md` 中依赖安装命令已统一为：

```bash
pip install -r scripts/requirements.txt
```

但 `check_env.py` 在依赖缺失时打印的安装提示仍为：

```bash
pip install -r requirements.txt
```

### 风险

- 用户可能在仓库根目录执行错误命令
- 当前项目真实依赖文件位于 `scripts/requirements.txt`，错误提示会误导执行路径
- 会削弱环境检查脚本作为主入口诊断工具的可信度

### 修复要求

后续修复时，应将失败提示中的依赖安装命令统一为：

```bash
pip install -r scripts/requirements.txt
```

### 验收标准

- `check_env.py` 的安装提示与 `SKILL.md` 一致
- 仓库根目录复制执行可直接生效

---

## 3.3 `check_env.py` 成功后的下一步提示口径不一致

### 问题描述

当前 `check_env.py` 成功后给出的下一步命令仍是旧口径，例如：

```bash
python extract_data.py --help
```

但当前 `SKILL.md` 的统一口径是：

```bash
python scripts/extract_data.py --help
```

### 风险

- 环境检查通过后，用户可能直接复制到错误路径口径的命令
- 导致脚本帮助提示与主文档不一致

### 修复要求

后续修复时，应统一成功后提示为仓库根目录运行口径。

### 验收标准

- `check_env.py` 成功后的下一步建议与 `SKILL.md` 保持一致

---

## 3.4 `extract_data.py` 顶部帮助文案仍保留旧运行方式

### 问题描述

当前 `extract_data.py` 顶部文档字符串中仍使用：

```bash
python extract_data.py --company=... --code=... --year=...
python check_env.py
```

而当前 `SKILL.md` 已明确默认推荐仓库根目录运行方式：

```bash
python scripts/extract_data.py --company=... --code=... --year=...
python scripts/check_env.py
```

### 风险

- 脚本自带帮助文案与主文档不一致
- 维护者会误以为旧目录口径仍是项目默认标准
- 容易在后续继续复制旧示例造成二次扩散

### 修复要求

后续修复时，应统一 `extract_data.py` 顶部帮助文案中的：

1. 主流程示例命令
2. `--help` 示例
3. 环境检查前置命令

全部改为与 `SKILL.md` 一致的仓库根目录口径。

### 验收标准

- `extract_data.py` 顶部文案与 `SKILL.md` 一致
- 不再保留旧的默认运行方式作为主示例

---

## 3.5 `extract_data.py` 中手动补救命令仍保留旧路径口径

### 问题描述

当前 `extract_data.py` 在表2生成失败时，提示用户手动执行 `generate_analysis_report.py`，但命令仍是旧口径，例如：

```bash
python generate_analysis_report.py ...
```

而当前主文档统一要求仓库根目录运行：

```bash
python scripts/generate_analysis_report.py ...
```

### 风险

- 失败补救路径会与主文档冲突
- 用户最需要帮助的“失败场景”反而得到错误示例

### 修复要求

后续修复时，应将 `extract_data.py` 中所有手动补救命令统一改为仓库根目录口径。

### 验收标准

- `extract_data.py` 终端提示里的补救命令与 `SKILL.md` 一致

---

## 3.6 `generate_analysis_report.py` 顶部“使用方法”与真实 CLI 不完全一致

### 问题描述

当前 `SKILL.md` 中给出的命令是完整参数形式：

```bash
python scripts/generate_analysis_report.py --company=公司名 --code=股票代码 --year=年份 --input=年报数据.xlsx --output=财报解读.xlsx
```

而 `generate_analysis_report.py` 顶部帮助说明若只保留简化版输入/输出示例，会与真实 CLI 参数要求不一致。实际脚本参数还要求：

- `--company`
- `--code`
- `--year`
- `--input`
- `--output`

### 风险

- 用户按顶部示例复制执行会失败
- 文档与真实 CLI 行为不一致
- 造成“脚本存在但示例不可用”的体验问题

### 修复要求

后续修复时，应将脚本顶部“使用方法”同步为完整 CLI 版本，并与 `SKILL.md` 保持一致。

### 验收标准

- `generate_analysis_report.py` 顶部示例参数完整
- 顶部示例与真实 `argparse` 要求一致
- 与 `SKILL.md` 示例保持一致

---

## 3.7 `SKILL.md` 的验收标准需向 `data_validator.py` 对齐，而不是反向放宽脚本

### 问题描述

当前 `data_validator.py` 的“可继续交付”判定明显严于 `SKILL.md` 中较宽松的最小验收标准。

`data_validator.py` 当前使用的更严格判定包括：

1. 文件存在
2. 文件可回读
3. 表1结构为 22 项
4. 完整率达到 100%
5. 数值问题为 0
6. 来源标注问题为 0
7. 来源不明项为 0
8. 五维度评分达到 70+

### 本次处理原则

本问题**不按“降低脚本规则”处理**。

本次修复需求明确要求：

> 以后续修正文档口径为主，让 `SKILL.md` 的验收标准向 `data_validator.py` 当前规则对齐。

### 风险

如果继续保留宽松版 `SKILL.md` 验收标准，会导致：

- 用户以为“表1、表2存在即可交付”
- 但脚本会判定“不可继续交付”
- 文档和脚本将长期存在双重标准

### 修复要求

后续应修改 `SKILL.md`，把“最小验收标准”升级为至少覆盖以下要点：

1. 表1文件存在
2. 表1可回读
3. 表1结构为 22 项
4. 表2文件存在
5. 表2可回读
6. 已执行验证
7. 数值问题已清零或已解释
8. 来源标注问题已清零
9. 来源不明项为 0
10. 五维度评分达到可继续交付阈值
11. `data_validator.py` 判定为可继续交付，才视为通过最终交付门槛

### 验收标准

- `SKILL.md` 与 `data_validator.py` 的交付口径一致
- 不再出现“文档说可以交付，但脚本说不能交付”的冲突

---

## 3.8 `smoke_test.py` 的定位需要补充边界说明

### 问题描述

当前 `SKILL.md` 中将 `smoke_test.py` 列为“离线烟测核心链路”，这一本身没有问题；但若未明确边界，执行者可能误解为：

- 烟测通过 = 真实目标公司的主流程交付已经通过

实际上，`smoke_test.py` 的本质是：

- 基于固定夹具验证本地生成链路
- 验证表1、表2、验证报告和结构链路是否通
- 不能替代真实 AKShare / 搜索链路 / 真实目标公司的完整交付验证

### 风险

- 执行者可能过度依赖离线烟测结果
- 误把本地链路可用当成真实数据交付可用
- 在真实任务里跳过对外部数据源与目标文件的实际检查

### 修复要求

后续应在 `SKILL.md` 中补充 `smoke_test.py` 的边界说明，例如：

1. 它是离线链路烟测，不是实际任务交付证明
2. 它验证的是本地生成链路，不等于外部数据源真实可用
3. 它不能替代真实公司、真实年份、真实双表落盘结果的验收

### 验收标准

- `SKILL.md` 对 `smoke_test.py` 的定位清楚
- 执行者不会把烟测通过误读成完整交付通过

---

## 3.9 脚本内说明性注释应保持克制

### 问题描述

当前部分脚本内存在一些偏“解释性”或“面向修改过程”的注释，虽不影响运行，但与当前项目的维护原则不完全一致。

本项目当前更强调：

- 注释解释非显然逻辑
- 不用注释重复代码表面含义
- 不把修改过程说明写进代码中

### 风险

- 后续脚本会继续累积“为当次修改服务”的注释
- 降低代码长期可维护性
- 与 `SKILL.md` 中的维护治理原则不一致

### 修复要求

后续如进行脚本整理时，应顺带清理以下类型注释：

1. 只是在描述“我现在要做什么”的注释
2. 只是在重复下一行代码字面含义的注释
3. 为临时修改而添加、但不解释长期逻辑的注释

### 验收标准

- 注释保留必要性、约束、非显然意图
- 删除过程性、口语化、重复性注释

## 7. 结论

本次《脚本修复需求文档》的核心结论是：

1. 当前关键脚本大体可用，但帮助文案、路径口径、验收口径仍存在不一致
2. 不应通过放宽 `data_validator.py` 的“可继续交付”规则来实现一致性
3. 应以 `data_validator.py` 当前更严格的规则为准，反向修正文档与说明口径
4. 修复重点应放在：
   - 命令示例统一
   - CLI 说明真实化
   - 验收标准统一
   - 烟测边界澄清

后续若据此实施修复，应优先修正文档和帮助文案，使其与现有脚本能力严格对齐。
